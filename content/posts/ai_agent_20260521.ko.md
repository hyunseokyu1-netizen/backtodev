• AI 에이전트는 “한 번 말하면 여러 단계를 스스로 진행해야 하는 일”에 가장 잘 맞
  습니다. 단순 질문 답변보다, 조사 → 판단 → 실행 → 검증까지 이어지는 작업에 쓰는
  게 좋습니다.

  잘 맞는 경우

  반복 작업:

  - 블로그 초안 정리
  - 커밋 로그 기반 작업 요약
  - 릴리즈 노트 작성
  - 매번 같은 형식의 문서 생성
  - 이미지/링크/frontmatter 검수

  코드 작업:

  - 버그 원인 찾기
  - 리팩터링 범위 조사
  - 테스트 실패 원인 분석
  - 작은 기능 구현
  - 여러 파일에 걸친 일관된 수정
  - 사용하지 않는 코드/의존성 정리

  조사와 정리:

  - 특정 라이브러리 사용법 확인
  - 경쟁 서비스/앱 기능 비교
  - 문서 읽고 요약
  - 프로젝트 구조 분석
  - “이 코드가 왜 이렇게 되어 있는지” 추적

  운영 자동화:

  - 배포 전 체크리스트 수행
  - SEO 메타데이터 검수
  - 예약 글 publish_date 확인
  - 깨진 링크 확인
  - 포트폴리오 카드 업데이트
  - 앱 스토어 설명/스크린샷 문구 정리

  잘 안 맞는 경우

  - 한 줄 답이면 되는 질문
  - 최종 판단이 취향이나 전략에 가까운 일
  - 실시간 민감 정보가 필요한데 확인 권한이 없는 일
  - 결제, 삭제, 배포처럼 실수 비용이 큰 일을 승인 없이 자동으로 하는 것
  - 요구사항이 아직 너무 모호한 일

  이런 경우에는 먼저 사람과 방향을 정하고, 그다음 에이전트에게 실행을 맡기는 게
  좋습니다.

  좋은 활용 방식

  에이전트에게는 “역할 + 입력 + 산출물 + 제한 조건”을 같이 주는 게 좋습니다.

  예:

  이 저장소에서 포트폴리오 페이지 구조를 분석해줘.
  수정은 하지 말고,
  1. 데이터가 어디에 있는지
  2. 이미지가 어디서 로드되는지
  3. 새 프로젝트 카드를 추가하려면 어떤 파일을 바꿔야 하는지
  정리해줘.

  또는:

  오늘 커밋을 분석해서 블로그 초안으로 정리해줘.
  대상은 git log 기준 오늘 커밋이고,
  초안은 ../blog_doc_temp/ 에 저장해줘.
  문체는 내 기존 한국어 포스트 스타일을 따라줘.

  실전 기준

  작업을 이렇게 나누면 좋습니다.

  - 내가 할 일: 목표, 우선순위, 최종 판단
  - 에이전트가 할 일: 탐색, 정리, 초안, 반복 수정, 검증
  - 사람이 승인할 일: 배포, 삭제, 결제, 공개 문구 확정, 민감 정보 변경

  지금 블로그 운영 기준으로는 특히 이런 에이전트가 유용합니다.

  - blog-auto-draft: 오늘 작업을 블로그 초안으로 변환
  - portfolio-updater: 프로젝트 md를 읽고 포트폴리오 카드/이미지 반영
  - post-checker: frontmatter, publish_date, 링크, 이미지 경로 검수
  - seo-reviewer: title/description/canonical/OG/sitemap 확인
  - release-note-writer: 앱 업데이트 내역을 스토어 문구로 정리

  핵심은 “에이전트에게 생각을 맡긴다”보다 “귀찮고 반복적인 중간 과정을 맡긴다”에
  가깝습니다. 최종 방향은 사람이 잡고, 에이전트는 속도와 누락 방지에 쓰는 게 가
  장 안정적입니다.
