## 앱 출시가 이렇게 복잡할 줄 몰랐다

예전엔 APK 파일 하나 올리면 바로 스토어에 올라가는 줄 알았다. 지금은 다르다. 구글 플레이는 신규 앱에 **비공개 테스트(Closed Testing)** 를 먼저 거치도록 유도하고, 테스트를 충분히 거친 앱에 한해 정식 출시 심사를 통과시켜 준다.

내가 만든 Tilt 게임도 그 과정을 지나고 있다. 최근에 버전을 올리면서 알게 된 게 있는데, **비공개 테스트 기간 중 업데이트를 자주 할수록 심사 통과 확률이 높아진다**는 것이다. 오늘은 그 경험을 정리해봤다.

---

## 비공개 테스트가 뭔데요

구글 플레이의 테스트 단계는 크게 세 가지다.

| 단계 | 설명 | 특징 |
|---|---|---|
| **내부 테스트** | 팀원, 본인만 접근 | 즉시 게시됨 |
| **비공개 테스트** | 초대된 사람만 접근 | 심사 있음, 일반 공개 전 단계 |
| **공개 테스트** | 누구나 참여 가능 | 정식 출시 전 마지막 단계 |

신규 앱은 구글이 "이 앱이 정말 운영될 의지가 있는가"를 확인하려 한다. 비공개 테스트는 그걸 증명하는 구간이다. 여기서 **최소 12명의 테스터가 14일 이상 앱을 사용해야** 다음 단계로 넘어갈 수 있다.

---

## 그런데 왜 업데이트 빈도가 중요한가

처음에 나는 빌드 하나 올려놓고 14일을 기다렸다. 근데 14일이 지나도 "프로덕션 액세스 신청" 버튼이 활성화되지 않았다.

이것저것 찾아보고 직접 겪어보니 이렇더라.

> **구글은 비공개 테스트 기간 동안 앱이 꾸준히 관리되고 있는지를 본다.**

단순히 "14일 + 12명" 조건만 맞춘다고 되는 게 아니다. 앱이 살아있다는 신호를 계속 보내줘야 한다. 그 신호 중 하나가 **업데이트 빈도**다.

버전을 올릴 때마다 구글 측에서 앱이 적극적으로 유지보수되고 있다는 걸 인식하는 것 같았다. 실제로 Tilt 게임도 버전을 몇 차례 올린 뒤로 심사 흐름이 눈에 띄게 달라졌다.

---

## Tilt 게임 버전업 실제 과정

### Step 1. 변경 사항 반영 후 `versionCode` 올리기

`build.gradle`(또는 `app.json`)에서 버전을 올린다. `versionCode`는 Play Store에서 업데이트 여부를 판단하는 기준이므로 반드시 이전보다 높아야 한다.

```json
// app.json 예시
{
  "expo": {
    "version": "1.3.0",
    "android": {
      "versionCode": 5
    }
  }
}
```

### Step 2. AAB 빌드

```bash
./build_aab.sh
# → /path/to/releases/tilt-v1.3.0.aab 자동 저장
```

### Step 3. 구글 플레이 콘솔에 업로드

1. [Google Play Console](https://play.google.com/console) 접속
2. 앱 선택 → **테스트 > 비공개 테스트**
3. **새 버전 만들기** 클릭
4. AAB 파일 업로드
5. 변경 사항 입력 후 **검토를 위해 출시** 클릭

### Step 4. 심사 대기

비공개 테스트도 심사가 있다. 보통 몇 시간에서 하루 이내로 처리된다. 내부 테스트와 달리 즉시 배포되지 않으니 기다려야 한다.

---

## 업데이트할 때 신경 쓰는 것들

### 변경 사항이 없어도 업데이트한다

버그가 없고 기능 추가가 없어도, 의미있는 개선이 생기면 버전을 올린다. 텍스트 수정, 성능 개선, UI 미세 조정도 충분한 이유가 된다.

### 릴리즈 노트를 성실하게 쓴다

```
v1.3.0
- 타이틀 화면 애니메이션 개선
- 점수 표시 오류 수정
- 진동 피드백 강도 조정
```

짧아도 되니까 매 버전마다 뭔가 변한 게 있다는 걸 보여주는 게 좋다.

### 테스터 피드백을 반영한다

비공개 테스트의 핵심은 테스터가 쓰고 피드백을 주는 것. 그 피드백을 다음 버전에 반영하면 자연스럽게 업데이트 주기가 생기고, 테스터도 더 적극적으로 참여하게 된다.

---

## 자주 하는 실수

**❌ 빌드 하나 올리고 2주 기다리기**
→ 구글 입장에서 "이 앱, 지금 관리되고 있나?" 싶어진다.

**❌ versionCode 올리는 걸 잊기**
→ 업로드 자체가 안 된다. 반드시 이전보다 높아야 함.

**❌ 테스터 수만 채우고 앱 방치**
→ 12명이 깔았어도 실제 사용 데이터가 없으면 조건 미충족.

---

## 정리

비공개 테스트는 단순히 기간을 채우는 게 아니다. 구글이 보는 건 이거다.

- ✅ 테스터가 실제로 앱을 쓰고 있는가
- ✅ 개발자가 꾸준히 앱을 관리하고 있는가
- ✅ 문제가 생겼을 때 대응하고 있는가

버전 업데이트는 그 모든 걸 보여주는 가장 직접적인 방법이다. 테스트 기간 중에는 완성도보다 **꾸준함**이 더 중요하다는 걸 Tilt 게임을 올리면서 직접 배웠다.

작은 수정이라도 버전을 올리고, 릴리즈 노트를 남기고, 테스터 피드백에 반응하는 것 — 이게 비공개 테스트를 빠르게 통과하는 현실적인 방법이다.
