# JobRadar 개발기 시리즈 — 작성 양식 가이드

1~5편 분석을 기반으로 정리한 스타일 가이드. 신규 편 작성 시 이 양식을 따를 것.

---

## Frontmatter

```yaml
---
title: '[JobRadar N편] 제목 — 부제목'
date: 'YYYY-MM-DD'          # 작성일
publish_date: 'YYYY-MM-DD'  # 배포 예정일 (scheduled 폴더용)
description: 한 문장 요약. 마침표 없이. 검색 결과에 표시되는 내용
tags:
  - JobRadar
  - 관련기술1
  - 관련기술2
  - 사이드프로젝트
---
```

**title 형식**: `[JobRadar N편] 제목` (1~2편 기준)
- 대시로 구분하는 부제목: `— 부제목` 형태
- 예: `[JobRadar 7편] Google OAuth "Database error" — 트리거 버그 찾기까지`

**tags**: `JobRadar`를 항상 첫 태그로, `사이드프로젝트`를 마지막으로.

## 도입부

H1 헤딩 없음. frontmatter의 title이 H1 역할.

**패턴 1** — 순서 목록으로 시작 (반복적인 루틴을 묘사할 때):
```markdown
호주/NZ IT 취업을 준비하면서 매일 반복하는 루틴이 생겼다.

1. Indeed 열기 → "React Native developer Sydney" 검색
2. Seek 열기 → 동일 검색
3. ...
```

**패턴 2** — 짧은 서술 + 인용구:
```markdown
3편까지의 이야기를 한 줄로 요약하면 이렇다.

> Playwright + Vercel = 3일 동안 에러만 봤다.
```

**패턴 3** — 짧은 서술만:
```markdown
지난 5편에서 URL을 붙여넣으면 JD 스크래핑 → AI 매칭까지 자동으로 돌아가는 파이프라인을 만들었다. 이번엔 그 위에 살을 붙이는 작업이다.
```

## 개요 섹션

### 만든 것들 한눈에 (기능 목록이 있을 때)

```markdown
## 만든 것들 한눈에

| 기능 | 내용 |
|------|------|
| 기능명 | 설명 |
```

### 전체 X 타임라인 (삽질 기록, 디버깅일 때)

````markdown
## 전체 삽질 타임라인

```
시도 1 → 결과
  → 시도 2 → 결과
  → 해결
```
````

---

## Step 섹션

**헤딩 형식**: `## Step N — 제목` (em dash `—` 사용, period `.` 아님)

```markdown
## Step 1 — URL 입력창과 플랫폼 자동 감지

본문 설명...

```typescript
// 코드 예시
```

---

## Step 2 — 다음 단계
```

- 각 Step 사이에 `---` 구분선
- 코드 블록 앞뒤로 간단한 설명 (코드만 던지지 않기)
- 긴 Step 안에 `### 소제목`으로 세분화 가능

---

## 트러블슈팅

`## 트러블슈팅` 헤딩 아래 에러/문제를 나열.

**형식 A** — bold 질문형 (짧은 설명일 때):
```markdown
## 트러블슈팅

**fetch가 403을 반환하는 경우**

User-Agent를 실제 브라우저처럼 설정하는 것만으로 상당수 해결된다.

**해결**:
```typescript
headers: { 'User-Agent': '...' }
```
```

**형식 B** — 증상/원인/해결 명시형:
```markdown
## 트러블슈팅

**Google 로그인 후 다시 로그인 페이지로 튕김**

미들웨어 matcher에서 `auth/callback`을 제외 안 했을 때 발생.

**해결**: matcher 패턴에 `auth/callback` 제외 추가.
```

**형식 C** — 표 요약 (에러가 많을 때):
```markdown
## 트러블슈팅 요약

| 에러 | 원인 | 해결 |
|------|------|------|
| `ETXTBSY` | /tmp 바이너리 동시 쓰기 | 순차 실행 |
```

## 정리 섹션 (마무리)

`## 정리 — 핵심 흐름 한눈에` 헤딩 하나에 코드블록 + 텍스트를 같이 담는다.

````markdown
## 정리 — 핵심 흐름 한눈에

```
전체 흐름을 ASCII 다이어그램으로
  ↓
단계별로 표현
```

마무리 텍스트. 이번 편의 핵심 교훈 또는 다음 편 예고.

다음 편에서는 X를 만들 예정이다.
````

## 전체 편별 비교 (1~7편)

| 편 | 개요 형식 | Step 형식 | 트러블슈팅 | 마무리 |
|----|----------|----------|----------|--------|
| 1편 | `## 기술 스택 한눈에 보기` (표) | `## Step N —` | bold 질문형 | `## 정리 — 핵심 흐름 한눈에` |
| 2편 | `## 기술 스택` (표) | `## Step N —` | 번호 목록 + 표 | `## 정리 — 핵심 흐름 한눈에` |
| 3편 | `## 전체 삽질 타임라인` (코드블록) | `## Step N —` | 표 | `## 정리 — 최종 구성 한눈에` |
| 4편 | `## 왜 바꿨나` + `## 구현 개요` | `## Step N —` | bold 질문형 | `## 정리 — 핵심 흐름 한눈에` |
| 5편 | `## 만든 것들 한눈에` (표) | `## Step N.` ⚠️ | `### 소제목` | `## 전체 파이프라인 한눈에` + `## 정리` |
| 6편 | `## 만든 것들 한눈에` (표) | `## Step N —` ✅ | bold 질문형 | `## 정리 — 핵심 흐름 한눈에` ✅ |
| 7편 | `## 전체 삽질 타임라인` (코드블록) | `## Step N —` ✅ | bold 질문형 | `## 정리 — 핵심 흐름 한눈에` ✅ |

> ⚠️ 5편은 Step에 period(`.`)를 사용해 1~4편과 다름. 6편부터는 em dash(`—`)로 통일.
